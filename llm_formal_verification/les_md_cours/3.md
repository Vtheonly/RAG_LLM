# Cours 3 : Annotations JML (Java Modeling Language)
**Auteur :** A. Giorgetti
**Pages / Diapositives :** 49 à 52

## Philosophie du JML
* Le JML est une extension de Java pour annoter le code source avec des spécifications formelles.
* Il permet la conception par contrats (*Design by Contract* - Meyer 1992) : 
  * Un contrat entre l'appelant (*caller*) et la méthode appelée (*callee*).
  * Le *callee* garantit un résultat si le *caller* garantit les prérequis.

## Exemple : Contrat de `sqrt`
```java
/*@ requires x >= 0.0;
  @ ensures JMLDouble.approximatelyEqualTo(\result * \result, x, eps); */
public double sqrt(double x) { ... }
```
| Rôle | Obligations | Droits |
| :--- | :--- | :--- |
| **Client** | Passe un nombre positif | Obtient la racine carrée |
| **Implémenteur** | Calcule la racine | Suppose x >= 0 |
```

# Cours 3 : Sémantique des contrats JML
**Auteur :** A. Giorgetti
**Pages / Diapositives :** 53 à 56

## Meaning of Pre-/Post-Condition Pairs
Une paire pré/post-condition pour une méthode `m` est satisfaite par l'implémentation si :
* À chaque appel de `m` dans un état satisfaisant la `précondition`, tout état terminal de `m` satisfait la `postcondition`.

## Remarques importantes
1. Aucune garantie n'est donnée si la précondition n'est pas respectée.
2. La terminaison n'est pas forcément garantie (sauf si `diverges` est spécifié).
3. Les états terminaux peuvent être des retours normaux ou des exceptions (abrupt termination).
```

### Fichier 15 : Cours 3 - Syntaxe JML Avancée
```markdown
# Cours 3 : Avancé JML (Assignable, Exceptions, Pure)
**Auteur :** A. Giorgetti
**Pages / Diapositives :** 69 à 74

## Clauses Assignables
La clause `assignable` liste les localisations pouvant être modifiées :
* `assignable loc1, ..., locn;`
* `\nothing` : rien ne peut changer.
* `\everything` : tout peut changer (défaut).

## Spécification de la Divergence
`diverges e;`
Spécifie que la méthode peut ne pas terminer uniquement si l'expression booléenne `e` est vraie dans l'état initial.

## Méthodes Pures (`pure`)
* Une méthode `pure` n'a aucun effet de bord et termine toujours.
* Peut être appelée dans les annotations JML.
* `pure = diverges false; + assignable \nothing;`

## Exceptions
Spécification avec `signals (E e) s;` pour définir ce qui doit tenir quand une exception de type `E` est levée.
```

### Fichier 16 : Cours 3 - Quantificateurs et Prédicats de Portée
```markdown
# Cours 3 : Expressions Quantifiées JML
**Auteur :** A. Giorgetti
**Pages / Diapositives :** 77 à 80

## Quantificateurs
* `(\forall t x; a; b)` : Pour tout `x` de type `t` satisfaisant `a`, `b` est vrai.
* `(\exists t x; a; b)` : Il existe un `x` de type `t` satisfaisant `a` tel que `b` est vrai.
* `a` est le **prédicat de portée** (*range predicate*).

## Exemples pragmatiques
* **Tableau trié :** `(\forall int i,j; 0 <= i && i < j && j < 10; a[i] <= a[j])`
* **Instance créée :** On restreint souvent la portée aux objets créés via le mot-clé `\created`.
* Les quantificateurs JML/KeY couvrent aussi les objets non-créés.
```

# Cours 4 : Logique du premier ordre (FOL)
**Auteur :** A. Giorgetti
**Pages / Diapositives :** 89 à 104

## Signature Σ
Une signature Σ consiste en :
* Un ensemble de types `TΣ`.
* Des symboles de fonctions `FΣ` et de prédicats `PΣ`.
* Une fonction de typage `αΣ` définissant l'arité et les types.

## Formules Atomiques
* `true`, `false`
* `t1 = t2` (égalité)
* `p(t1, ..., tn)` (prédicat)

## Formules Générales
Induction : `¬φ`, `φ ∧ ψ`, `φ ∨ ψ`, `φ → ψ`, `φ ↔ ψ`, `∀τ x; φ`, `∃τ x; φ`.
* Les variables liées par un quantificateur ne sont pas libres (formules "fermées" si aucune variable libre).
```

### Fichier 18 : Cours 4 - Sémantique et Calcul des Séquents
```markdown
# Cours 4 : Sémantique et Calcul des séquents
**Auteur :** A. Giorgetti
**Pages / Diapositives :** 109 à 144

## Sémantique
* L'interprétation nécessite une signature, un domaine (univers typé `D`), et une fonction d'interprétation `I`.
* La validité (`|= φ`) est vraie si la formule est vraie dans tous les modèles (états).

## Calcul des Séquents
Utilisé pour prouver la validité de manière syntaxique (sans énumérer les modèles) :
`Γ =⇒ Δ`
* **Axiomes :** `Γ, φ =⇒ φ, Δ` est toujours valide.
* **Règles :** Permettent de transformer les formules (`andRight`, `orLeft`, `allRight`, etc.).
* **Soundness & Completeness :** Le système est correct (si une preuve close existe, la formule est valide) et complet (si une formule est valide, une preuve close existe).
* **Preuve de `∃` ou `∀` :** Nécessite des règles d'instanciation (souvent avec des constantes "nouvelles" pour `allRight` ou des termes témoins pour `exRight`).
```
