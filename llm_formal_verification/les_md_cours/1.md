## Logique de Hoare
La Logique de Hoare permet de raisonner sur la **correction partielle** d'un programme impératif par rapport à une **précondition** et une **postcondition**.
* C'est une logique de programmes.
* C'est un système formel basé sur des règles d'inférence.
* Autre nom : **Sémantique axiomatique**.
* **Référence historique :** C. A. R. Hoare, *An axiomatic basis for computer programming*, Commun. ACM, 1969.

## Mini-langage de programmes impératifs
Le langage étudié est un langage "tant-que" (Turing-complet) qui inclut les instructions (`c`) suivantes :
```text
c, c' ::= | x := e                           (affectation)
          | c; c'                            (séquence)
          | if b then c else c' fi           (condition)
          | while b do c od                  (répétition)

x est une variable, e est une expression, b est une expression booléenne.

Les mots-clés fi et od évitent les confusions d'interprétation pour les commandes imbriquées.

Comportements possibles : terminaison ou divergence (boucle infinie).

Triplet de Hoare

Un triplet de Hoare s'écrit {P} c {Q}.

c : la commande (instruction).

P : la précondition (propriétés d'état avant exécution).

Q : la postcondition (propriétés d'état après exécution).

Validité : Le triplet {P} c {Q} est valide si, lorsque P est satisfaite, la commande c (si elle termine) établit la postcondition Q.

Note : On parle de correction partielle car on ignore les cas de non-terminaison (divergence).

La logique de Hoare est un système déductif correct et complet.

## Système de règles d'inférence (Sémantique axiomatique)
Le système déductif repose sur plusieurs règles permettant de prouver la validité des triplets de Hoare :

### 1. Règle pour l'affectation
```text
{P (e/x)} x := e {P}    (assignment)

C'est un axiome (il n'y a pas de prémisse).

(e/x) formalise la substitution uniforme de la variable x par l'expression e dans la formule P.

L'affectation x := e donne la valeur de e à x. Donc P (portant sur x) est vraie après l'affectation si P(e/x) était vraie avant.

2. Règle pour la composition séquentielle
code

download
content_copy
expand_less
{P} c {Q}      {Q} c' {R}
------------------------------ (sequence)
        {P} c; c' {R}

Pour dériver {P} c; c' {R}, il faut deviner une formule intermédiaire Q.

Q agit comme une notion d'assertion pour documenter le programme.

Le calcul de Q amène à la notion de plus faible précondition.

3. Règle pour l'instruction conditionnelle (if-then-else)
code

{b ∧ P} c1 {Q}      {¬b ∧ P} c2 {Q}
------------------------------------------ (if-then-else)
    {P} if b then c1 else c2 fi {Q}

L'expression booléenne b est considérée comme une formule portant sur les variables d'état.

Si b n'est pas une formule stricte, on la remplace par b = true et ¬b par b = false.

## Règle pour l'instruction itérative (while)
```text
          {b ∧ P} c {P}
---------------------------------- (while)
 {P} while b do c od {¬b ∧ P}

P est appelé invariant de boucle.

Interprétation : Si l'invariant P est vrai au début (précondition) et si une itération préserve P (prémisse {b ∧ P} c {P}), alors à la fin de la boucle, l'invariant P est toujours vrai et la condition de boucle b est fausse (¬b ∧ P).

Difficulté : Il faut deviner cet invariant P (synthèse d'invariant).

Règle de conséquence

               {P} c {Q}

Cette règle sert à "recoller les morceaux" pour adapter une précondition ou postcondition stricte.
Cas typique d'utilisation avec while :

P ⇒ I     {I} while b do c od {¬b ∧ I}     ¬b ∧ I ⇒ Q
----------------------------------------------------------
                {P} while b do c od {Q}
Exemple de preuve en logique de Hoare (1/2)

Objectif : Prouver la factorielle r = n!.
Programme : i := 0; r := 1; while i < n do i := i + 1; r := r * i od
Précondition : {n ≥ 0}
Postcondition : {r = n!}

Trouver l'invariant I : L'invariant approprié avant et pendant la boucle est I ≡ (i ≤ n ∧ r = i!).

Prouver que l'invariant est vrai avant la boucle :
n ≥ 0 ∧ i = 0 ∧ r = 1 ⇒ i ≤ n ∧ r = i! (ce qui est vrai car 0 ≤ n et 1 = 0!).

Prouver qu'une itération préserve l'invariant :
Démontrer le triplet : {i < n ∧ r = i!} i := i + 1; r := r * i {i ≤ n ∧ r = i!}

## Exemple de preuve en logique de Hoare (Suite et fin)
L'étape finale de la preuve de factorielle consiste à montrer que la sortie de la boucle implique la postcondition finale :
* **Négation de la condition de boucle + Invariant ⇒ Postcondition**
* `¬(i < n) ∧ (i ≤ n ∧ r = i!) ⇒ r = n!`
* En effet, `¬(i < n) ∧ i ≤ n` implique `i = n`. En remplaçant `i` par `n` dans `r = i!`, on obtient `r = n!`. CQFD.

## Annotation du code par des assertions
Plutôt que d'écrire de grands arbres de preuve, on peut annoter directement le code avec des assertions entre chaque instruction :
```text
{n ≥ 0}
i := 0;
r := 1;
{i ≤ n ∧ r = i!}  // Invariant de boucle deviné
while i < n do
  {i < n ∧ i ≤ n ∧ r = i!}
  i := i + 1;
  r := r * i
  {i ≤ n ∧ r = i!}
od
{i ≮ n ∧ i ≤ n ∧ r = i!}
{r = n!}
Assertions principales et conditions de vérification

L'approche moderne de la preuve consiste à :

Extraire les obligations de preuves grâce aux annotations (invariants, pré/postconditions).

Démontrer chaque petit sous-arbre de manière modulaire (ex: la préservation de l'invariant, et la condition de sortie).
