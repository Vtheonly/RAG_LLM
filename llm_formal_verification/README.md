# Learning-Infused Formal Reasoning: Local LLM & RAG for Why3 Proof Synthesis

## 1. Project Context & Academic Objectives
This repository contains the experimental software architecture for the Master 2 Research Project (UMLP, Besançon, France) belonging to the **VESONTIO** team. 

**Research Question:** 
*To what extent can a local, small-scale Large Language Model (Gemma 4:E2B), coupled with a Retrieval-Augmented Generation (RAG) architecture, aid in formally proving a program using an expert's course materials?*

### 1.1 Core Research Inquiries
1. **Hallucination Mitigation:** Does forcing the AI to retrieve context from highly technical PDF course slides prevent it from inventing false mathematical rules?
2. **Local CPU Viability:** Can a highly quantized, small-parameter model (2B parameters) running entirely locally on a CPU comprehend formal logic rules and apply them to novel code?
3. **Failure Boundaries:** At what level of mathematical complexity (e.g., loops, invariants, Euclidean division) does the model's reasoning collapse?
4. **Explicability & Traceability:** Are the errors generated by the AI easily identifiable by the human user through automated verification feedback?

## 2. Architectural Design & Philosophy
To rigorously test these hypotheses, the codebase has been engineered using **Clean Architecture** and **SOLID design patterns**. This ensures the system is modular, easily extensible for future LLMs, and highly traceable via integrated logging.

### 2.1 Subsystems Breakdown
* **`config.py`**: Centralized configuration management (paths, hyper-parameters).
* **`data` module**: 
  * `document_loader.py`: Ingests raw PDFs from `./les cours`. Implements a robust text-cleaning pipeline to sanitize academic slide artifacts (page numbers, hyphenation) that typically degrade RAG performance.
  * `rag_manager.py`: Handles ChromaDB vectorization and similarity search using HuggingFace embeddings.
* **`llm` module**: 
  * `model_manager.py`: Abstracts the hardware layer. Loads **Gemma 4:E2B** via `kagglehub` and `transformers`. Enforces `bfloat16` and `inference_mode` to allow execution on consumer-grade CPUs without out-of-memory errors.
* **`verification` module**: 
  * `why3_runner.py`: The interface between the LLM and the formal world. Extracts MLW code using Regex, normalizes module wrappers, and spawns localized subprocesses to run the `alt-ergo` SMT solver via Why3.
* **`agent` module**:
  * `prover_agent.py`: Implements the iterative "Agentic Loop". It acts as the orchestrator, querying the LLM, passing the result to the Verifier, reading the `stderr/stdout` feedback, and prompting the LLM to self-correct up to $N$ times.
* **`evaluation` module**: 
  * `evaluator.py`: Standardizes the benchmarking process across three methodologies: Baseline (Zero-shot), RAG (Context-injected), and Agentic RAG (Context + Self-Correction).

## 3. Installation & Setup

### 3.1 System Prerequisites
Ensure you have the OPAM package manager, Why3, and Alt-Ergo installed on your local Linux/WSL environment:
```bash
sudo apt-get update
sudo apt-get install -y opam m4 build-essential libgmp-dev why3 alt-ergo
```

### 3.2 Python Environment
It is highly recommended to use a virtual environment.
```bash
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
```

### 3.3 Course Materials
Ensure that the expert course materials (PDFs) are located in the `les_cours/` directory at the root of the project.

## 4. Execution
To run the automated evaluation pipeline:
```bash
python main.py
```
The system will:
1. Initialize the logger.
2. Parse and vectorize the PDFs.
3. Download (if necessary) and load the Gemma 4 model into CPU RAM.
4. Run the benchmark tests and output a Pandas DataFrame and a performance plot.
